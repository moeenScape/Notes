Google Java Style Guide with Universal Examples

This page summarizes the Google Java Style Guide with simple, universal examples that any Java developer can understand. It covers coding standards for consistency and readability, with correct and incorrect code snippets.

1. Introduction

The Google Java Style Guide defines mandatory coding standards for Java in Google’s open-source projects, ensuring consistent, readable code.

Key Rule: All Java code must follow these rules to be “in Google Style.”

Example: A simple calculator app should use these standards for clean code.

2. Source File Basics

Rules:

  • File name: Matches top-level class, UpperCamelCase, `.java` extension.
  • File encoding: UTF-8.
  • Structure: License, package, imports (no wildcards), one top-level class.

Example: A user data class file.


// User.java
package com.example.app;

import java.util.List;

public class User {
    private String username;
}
                

// user_data.java (Wrong naming)
import java.util.*; // Wildcard import

public class userData { ... }

3. Formatting

Rules:

  • Braces: K&R style, always used, even for one-liners.
  • Indentation: 2 spaces, no tabs.
  • Line length: 100 characters max.
  • Whitespace: Spaces around operators, no spaces before parentheses.

Example: A method to check if a number is positive.


public boolean isPositive(int number) {
  return number > 0;
}
                

public boolean isPositive(int number)
{ // Wrong brace placement
    return number>0; // No spaces
}

4. Naming

Rules:

  • Packages: lowercase, no underscores (e.g., com.example.app).
  • Classes: UpperCamelCase (e.g., Calculator).
  • Methods: lowerCamelCase, verbs (e.g., calculateSum).
  • Variables: lowerCamelCase (e.g., totalAmount).
  • Constants: UPPER_SNAKE_CASE (e.g., MAX_VALUE).

Example: A calculator class.


package com.example.app;

public class Calculator {
    private static final int MAX_VALUE = 1000;

    public int calculateSum(int firstNumber, int secondNumber) {
        int totalAmount = firstNumber + secondNumber;
        return totalAmount;
    }
}
                

package com.example.APP; // Wrong case

public class calculator { // Wrong naming
    private static final int maxValue = 1000; // Not UPPER_SNAKE_CASE

    public int Calculate_Sum(int FirstNumber, int SecondNumber) { // Wrong naming
        int Total_Amount = FirstNumber + SecondNumber; // Wrong naming
        return Total_Amount;
    }
}

5. Programming Practices

Rules:

  • @Override: Always use when overriding.
  • Exceptions: Never ignore caught exceptions; log or rethrow.
  • Static members: Access via class name, not instance.
  • Finalizers: Avoid finalize().

Example: File reading utility.


import java.io.IOException;
import java.util.logging.Logger;

public class FileReader {
    private static final Logger LOGGER = Logger.getLogger(FileReader.class.getName());

    @Override
    public String toString() {
        return "FileReader";
    }

    public void readFile(String path) {
        try {
            FileReader.process(path);
        } catch (IOException e) {
            LOGGER.severe("Failed to read file: " + e.getMessage());
        }
    }

    private static void process(String path) throws IOException { ... }
}
                

public class FileReader {
    public String toString() { // Missing @Override
        return "FileReader";
    }

    public void readFile(String path) {
        try {
            process(path);
        } catch (IOException e) {} // Ignored exception
    }

    private void process(String path) throws IOException { ... } // Instance method for static
}

6. Javadoc

Rules:

  • Required for public classes/methods, starts with a summary.
  • Use @param, @return, @throws in order.
  • 80-character summary, one blank line between paragraphs.

Example: Documenting a string utility method.


/**
 * Checks if a string is empty or null.
 *
 * @param input the string to check
 * @return true if the string is empty or null, false otherwise
 * @throws IllegalArgumentException if input is invalid
 */
public boolean isEmptyString(String input) {
    return input == null || input.isEmpty();
}
                

/** Checks string */ // Too vague, missing tags
public boolean isEmptyString(String input) {
    return input == null || input.isEmpty();
}

7. Exceptions

Rule: Handle exceptions clearly, aligning with Section 5.2 (don’t ignore).

Example: Validating user input.


public void validateUserInput(String username) {
    if (username == null || username.isEmpty()) {
        throw new IllegalArgumentException("Username cannot be null or empty");
    }
}
                

public void validateUserInput(String username) {
    if (username == null || username.isEmpty()) {
        return; // Silently ignores error
    }
}
